# Story Document Format

## Overview

Each reconstruction entry is centered around a single file:


story.md


This document serves two purposes:

1. It provides structured metadata used by the system.
2. It records the Archivist's narrative explanation of the reconstruction.

The file combines **YAML frontmatter** and **Markdown text**.

This format is widely used in documentation systems and static site generators because it is simple, human-readable, and easy for software tools to parse.

---

## File Structure

The document is divided into two sections.

YAML metadata

Markdown narrative text


The YAML section contains structured fields used by tools.  
The Markdown section contains the human narrative.

---

## Example


entry_id: entry-0000003
created_utc: 2026-03-06 07:57:36+00:00

location_text: Australia/Sydney/Bondi Beach
year: 2027
datetime: 2027-01-01

status: Draft
tags: []

base_map: basemap.png

images: {}
generated_assets: {}

px_to_m: [1.0, 1.0]
map_orientation_deg: 0

base_map_centre_px: [0, 0]
main_feature_centre_px: [0, 0]

main_feature_gps:
lat: 0
lon: 0
Title

Write the story here.

## Metadata Fields

### entry_id

The unique identifier of the entry.

Example:


entry-0000003


This corresponds to the directory name containing the entry.

---

### created_utc

The UTC timestamp when the entry was created.

Example:


2026-03-06 07:57:36+00:00


---

### location_text

Human-readable location description.

Example:


Australia/Sydney/Bondi Beach


This field is descriptive and does not need to follow strict geographic standards.

---

### year

The historical year represented by the reconstruction.

Example:


1857


---

### datetime

Full ISO-8601 datetime representation.

Example:


1857-01-01


This allows precise ordering of historical entries.

---

### status

Current working state of the entry.

Typical values:


Draft
Review
Published


---

### tags

Optional classification tags.

Example:


tags:

colonial

port

singapore


---

### base_map

The primary image used as the reconstruction basemap.

Example:


basemap.png


---

### images

A dictionary of referenced image files.

Example:


images:
coastline_overlay: coastline.png


---

### generated_assets

References to files generated by tools.

Example:


generated_assets:
ground_mesh: assets/ground.glb


---

## Map Calibration Fields

Several fields describe how the basemap should be interpreted spatially.

These fields allow the system to align images with reconstructed geometry.

---

### px_to_m

Defines how many meters correspond to one pixel.


px_to_m: [1.0, 1.0]


Values represent:


[x_scale, z_scale]


---

### map_orientation_deg

Clockwise rotation applied to the basemap.


map_orientation_deg: 0


This allows maps drawn with rotated orientation to be aligned with the world coordinate system.

---

### base_map_centre_px

Pixel coordinate that should map to the world origin.


base_map_centre_px: [0, 0]


This controls how the map is centered when imported.

---

### main_feature_centre_px

Pixel location of a known landmark or reference feature.


main_feature_centre_px: [0, 0]


This can be used to align multiple maps across time.

---

### main_feature_gps

Real-world coordinates corresponding to the reference feature.


main_feature_gps:
lat: 0
lon: 0


This allows the reconstruction to be related to modern geographic data.

---

## Narrative Section

After the metadata block, the document continues with Markdown text.

This section records the Archivist's explanation of the reconstruction.

Typical contents include:

- historical background
- interpretation of sources
- explanation of modeling choices
- notes on uncertainty

Because this section is free-form Markdown, it can contain:

- headings
- images
- links
- tables
- code blocks

---

## Language Support

The document is stored in UTF-8 encoding.

This allows the narrative and metadata values to contain:

- English
- Hebrew
- Japanese

The system does not require a language marker within the file.

---

## Editing

The file can be edited using any text editor, including:

- Notepad
- VS Code
- Vim
- Emacs

Because the format is plain text, it remains durable and portable.

---

## Importance of the Story File

The `story.md` document is the **central artifact of the system**.

It connects:

- historical narrative
- map calibration
- generated geometry
- exported visualizations

If an entry directory is preserved, the reconstruction can always be understood by reading this file.
